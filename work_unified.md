# 1부 — 전체 그림 (왜 / 무엇)

## 1. 이게 뭔지

**aaa 프로젝트 (Auto Auto Auto — 자동작성·자동검토·자동수정)**

사용자의 주문에 따라 무료 모델(26B/31B)이 멀티파일 프로그램을 설계·코딩·검토·도커실측·출력하는 시스템. **의미 있는 초기 프로토타입을 먼저** 얻는 게 목표이고, 거기에 점점 살을 붙여 개량해 나가는 방향. 전통적인 “기획서 작성 → 프로토타입 제작”이 아니라, **어느 정도 짜인 초기 프로토타입을 빠르게 얻어 개량**하는 게 핵심.

## 3. 모델

- **전 역할 단일 모델 `gemini-3.5-flash`** (planner/coder/scripter/client). 정본 최초의 26B/31B 분업 없음.

## 4. 핵심 흐름 (전체 그림)

> ★ 아래 (26B)/(31B)는 **기능 역할명**이지 별도 모델 아님. 현재 전 역할 gemini-3.5-flash 단일(3장). 역할 분리는 H2 보류로 살려둔 기록 — 재분리 시 model 인자만 다르게.

두 모델이 티키타카하며 요구사항 → 배포 묶음까지 만드는 한 바퀴. **되돌림(피드백)이 양방향인 게 핵심** — 회사처럼, 코딩하다 설계 결함을 발견하면 위(설계)로 신고하고, 검토·실행에서 드러난 문제도 원인에 따라 설계/코드로 되돌린다.

```
① 설계 (26B)         파일 구조 + 각 파일 역할 + 인터페이스(시그니처) [JSON]
      ↓
①.5 설계 검토 (31B)   코딩 전 검문소: 구현 가능한가? 모순 없나?
      ├─ 불가/모순 → ① 로 되돌림
      ↓ (OK)
② 파일별 코딩 (31B)    파일 하나씩 (병렬 가능하나 순차로도 충분 — 6장)
      ↓
③ 정합성 검토 (26B)    코드 vs 설계 대조: A가 부른 함수를 B가 정의했나?
      │               ★ 계단식 검증 파이프라인으로 확장(8장)
      ↓ (통과)
④ 도커 실측           격리 컨테이너에서 실제 실행 → 성공/실패·에러·출력
      │               되돌림 판정(26B): 설계 문제 → ① / 코드 문제 → ②
      ↓ (통과)
⑤ 출력               전체 파일 + requirements + 실행법 묶음
```

이 흐름의 두 핵심:

1. **①이 “단일 파일”이 아니라 “파일 구조 + 인터페이스 설계”** — 멀티파일 정합성이 여기서 결정된다(5장 컨텍스트 관리와 직결).
1. **④ 도커 실측이 “배포 가능”을 추측이 아니라 검증으로 만든다** — 진짜 에러를 받아 26B가 원인을 판정한다(7장 도커).

되돌림 판정(“설계냐 코드냐”, 애매하면 코드 먼저)은 ③·④에서 26B가 갈라 보낸다. **③의 계단식 검증 파이프라인과 에러 3계층 분류(설계/코드/환경)는 8장**에서, 안쪽 자동·바깥 사람의 2층 반복은 9장에서 다룬다.

## 5. 컨텍스트 관리 — 시그니처만 공유 (멀티파일의 급소)

정합성의 성패가 여기서 갈린다. **방식: 인터페이스(시그니처)만 공유.** 파일 B를 짤 때 다른 파일들의 **전체 코드가 아니라 함수 시그니처만** 넣는다.

- **회사 비유**: 동료 코드를 다 읽는 게 아니라 API 문서(함수 명세)만 보고 짜는 것.
- **이유**: 멀티파일 정합성의 약점이 “긴 입력에서 놓치기”인데, 시그니처만 넣으면 **입력이 짧게 유지**돼 그 약점을 정면 회피한다. 파일 수십 개여도 시그니처 몇 줄이면 된다.
- **대가**: ① 설계가 인터페이스를 정확히 박아야 한다 → 이 위험은 ①.5(설계검토)와 ④(도커 실측)가 잡는다.

이걸 가능케 하는 게 ①의 설계 JSON이다. `provides`(이 파일이 제공하는 시그니처)와 `depends_on`(누구를 import하는지)이 멀티파일 정합성의 뼈대.

### 시그니처만으론 부족 — 데이터 계약(input/output_schema)까지

이름이 맞아도 **데이터 모양이 어긋나면 런타임에서 깨진다.** `save_user(user) -> bool`에서, 부르는 쪽은 `{"name": str}` dict를 넘기는데 받는 쪽은 `User(name, email)` 객체를 기대하면 — 시그니처는 똑같이 `save_user(user)`라 ①.5·③의 이름 대조를 통과하지만 도커에서 터진다. 멀티파일 실패의 큰 몫이 함수 이름이 아니라 이 **데이터 계약** 불일치다.

그래서 `provides`의 각 시그니처에 **input_schema·output_schema**(인자/반환의 타입·모양)를 함께 적는다. 단, **여기까지가 상한선이다 — 도메인 로직·구현 규칙은 JSON에 넣지 않는다**(1.2의 JSON DSL화 위험). 스키마는 “데이터가 어떻게 생겼나”(Data Contract)만, “무엇을 달성하나”는 acceptance_criteria(Domain Contract)가, 그 이상의 구현 방법은 코드가 가진다.

### 설계 JSON 형식 (① 출력 — 26B에게 이 스키마로 내라고 지시)

```json
{
  "assumptions": ["저장은 로컬 JSON 파일", "UI는 CLI(웹 아님)", "할 일 한 건은 {text:str, done:bool} dict"],
  "files": [
    {
      "name": "storage.py",
      "role": "할 일 데이터를 파일에 저장/불러오기",
      "provides": [
        {"sig": "save_todos(todos)", "input_schema": "list[ {text:str, done:bool} ]", "output_schema": "bool"},
        {"sig": "load_todos()",      "input_schema": "()",                            "output_schema": "list[ {text:str, done:bool} ]"}
      ],
      "depends_on": []
    },
    {
      "name": "todo.py",
      "role": "할 일 추가/삭제/완료 처리",
      "provides": [
        {"sig": "add_todo(todos, text)",     "input_schema": "(list, str)", "output_schema": "list[ {text:str, done:bool} ]"},
        {"sig": "complete_todo(todos, idx)", "input_schema": "(list, int)", "output_schema": "list[ {text:str, done:bool} ]"}
      ],
      "depends_on": ["storage.py"]
    },
    {
      "name": "main.py",
      "role": "CLI 진입점",
      "provides": [
        {"sig": "main()", "input_schema": "()", "output_schema": "None"}
      ],
      "depends_on": ["todo.py", "storage.py"]
    }
  ],
  "entry_point": "main.py",
  "acceptance_criteria": ["할 일을 추가하면 목록 길이가 1 늘어난다",
                          "완료 처리하면 해당 항목의 done이 True가 된다",
                          "삭제하면 목록에서 사라진다"]
}
```

- **`assumptions`**: 요구사항이 불명확해 26B가 메운 가정들. 사람이 가능성 판단할 때 “멋대로 했나”가 바로 보이고(9장), 가정이 틀렸으면 요구사항을 다듬어 재투입하는 근거가 된다.
- **`acceptance_criteria`**: 요구사항이 *달성*해야 할 것(Domain Contract — 무엇을 해야 하나). `provides`의 input/output_schema가 *어떻게 생겼나*(Data Contract)라면, 이건 *무엇을 해야 하나*다. **단 “절대적 검증 기준”이 아니라 “테스트 생성을 위한 힌트”다** — AI가 criteria만 보고 통과용 빈 껍데기 테스트(예: `assert search("") is not None`)를 쉽게 만들 수 있어서, 이것만으로 “성공”을 선언하지 않는다. 그래서 **지금(MVP)은 스키마에 필드만 둔다**(0원). 이 필드로 “criteria 통과하는 pytest를 자동 생성”까지 돌리는 건 **P1**으로 미룬다(13장) — 그때도 최종 품질 판정은 사람이 한다(9장 2층). JSON을 프로그래밍 언어(DSL)로 만들지 않으면서 시스템이 ‘요구사항 충족 여부’를 1차로 거들게 하는 상한선이다.
- ② 코딩 시 31B에게는 **“네가 짤 파일 명세 + 다른 파일들의 시그니처(코드 아님)”만** 준다. 26B 프롬프트엔 “구조만, JSON만, 마크다운 백틱 금지”를 박는다(파싱 안정).

*(부수 이득: 입력 토큰이 보통 API 비용의 큰 몫이라, 유료 모델로 일부 갈아탈 때도 시그니처만 공유가 그대로 정답이다.)*

## 6. 한도 회피 전략 — 공유 슬롯 관리자

한도 숫자는 3장 참조(RPM 15 / RPD 1,450 안전선 / TPM 무제한). 전략의 핵심: **TPM이 무제한이라 token bucket(분당 토큰 예측)은 불필요하다.** 남은 RPM·RPD는 둘 다 “횟수”라, 토큰이 아니라 **호출 횟수 슬롯**만 관리하면 된다.

```
공유 슬롯 관리자 1개:
  ├─ 최근 60초 호출 타임스탬프 → 15개 미만일 때만 통과 (RPM)
  ├─ 오늘 누적 호출수(usage.json, PT 자정 리셋) → 1,450 안전선 (RPD)
  └─ 429 맞으면 → 지수 백오프 후 재시도 (표시값 어긋남 대비 안전망)

호출 직전에 워커가 관리자에게 "슬롯 있어?"를 묻는다.
TPM 무제한이라 토큰 경쟁이 없고, 호출 횟수 슬롯만 공유 → 자연히 줄 선다.
```

- **주목적은 한도 준수**(429 방지·RPD 안전선 사수). 토큰 예측이 아니라 “비어있는 호출 슬롯을 나눠 쓰기”라 생각보다 단순하다.
- **병렬은 부수 효과**: 같은 슬롯 관리자를 워커 여러 개가 공유하면 병렬도 가능하다. 단 aaa의 목적은 속도가 아니라 싸게·안정적으로 토대를 뽑는 것이라 **순차로 돌려도 충분하다**(4장 ②). 병렬을 쓰든 안 쓰든 한도는 이 관리자가 지킨다.
- **대시보드 표시값과 실제가 어긋나 429가 나는 사례가 보고됨** → 무제한을 믿되 429는 반드시 잡는다(백오프).
- **RPD 안전선 초과 시**: 자정까지 대기는 자동 루프에서 위험하니, **멈추고 사람이 본다**(가능성 단계 정책). 멈춤은 디스코드 알림으로 폰에 통지된다(10장).

이 공유 슬롯 관리자가 aaa 한도 부분의 핵심이다. (구현은 토대의 한도 슬롯 모듈)

## 7. 도커 — 실측 실행 (작업 명세)

aaa의 핵심 가치는 **실측**이다. 생성된 멀티파일 프로그램을 격리 컨테이너에서 진짜로 돌려, 실행 성공/실패·에러·출력을 받는다. 그래야 “배포 가능”이 추측이 아니라 검증이 되고, 26B의 디버깅 판정도 실측 에러를 연료로 삼는다.

**도커가 할 일은 사실 단순하다: exit code / stdout / stderr 받고 → 컨테이너 폭파.** 기계적이고 환각이 없다.

### 왜 VM 직접 실행이 아니라 도커인가

1. **안전** — AI가 생성한 코드가 이상한 짓을 해도 컨테이너만 버리면 VM은 멀쩡.
1. **청결** — 매번 새 빈 환경. 이전 실행 찌꺼기가 없다.
1. **재현성** — 같은 환경 → 결과를 신뢰할 수 있다.

### 블랙박스 계약 (★ 확정 — 도커 본문을 나중에 채워도 이 입출력은 불변)

`runner.py`의 `run_in_docker`가 지키는 계약. 검토(8장)와 출력 루프는 이 모양에만 의존하므로, 도커 알맹이를 나중에 채워도 그쪽 코드는 안 바뀐다.

```
[입력]
  codes        : {파일명: 코드문자열}   (③ 코딩 출력)
  requirements : 설치할 라이브러리 목록
  entry_point  : 실행 시작 파일 (설계 JSON의 entry_point, 예 'main.py')
  timeout      : 실행 제한 시간(초). 입력 대기/무한루프 방어.

       ↓ (컨테이너: 빈 공간 → 파일 넣기 → 라이브러리 설치 → 실행 → 지켜보기 → 폭파)

[출력] dict:
  {
    "success":   bool,            # 에러 없이 의도대로 실행됐나
    "exit_code": int,
    "stdout":    str,             # 프로그램 출력
    "stderr":    str,             # 에러 메시지 (← judge_error의 연료)
    "timed_out": bool,
    "stage":     "build"|"run"    # 어디서 실패했나(라이브러리 설치 중 vs 실행 중)
  }
```

`stderr`가 자동 디버깅 루프의 연료다. 진짜 에러를 받아 26B에 주고 “설계/코드/환경”을 판정한 뒤 고친다(에러 3계층 분류 장 참조).

### “에러 없이 종료 ≠ 정상 동작” 함정

도커가 success를 줘도 그게 정상 동작이라는 보장은 없다. 빈 프로그램·삼킨 예외·입력 대기 후 즉시 종료도 success로 보일 수 있다. → `looks_alive(run_result, expected)`로 한 겹 더 본다: 기대 출력(expected) 일부가 stdout에 실제로 찍혔는지 확인. 이게 가능성 1층(자동 최소선) 판정의 보강이다.

*(바닥선 테스트와 합쳐져야 이 함정이 막힌다 — 8장 참조. “기능 함수를 호출하고 반환값을 assert”하는 테스트가 success의 빈 껍데기를 거른다.)*

### 현실 / 범위

- 가능성 단계(컨테이너 1개로 작은 파이썬 프로그램)는 e2-small로 충분히 가볍다.
- 여러 컨테이너 동시 실행(병렬 테스트)은 가능성 확인 뒤 걱정할 일(13장).
- 파이썬에서 도커를 다루는 법(subprocess vs docker SDK)은 구현 시 결정 — 어느 쪽이 단순한지는 실제로 짜보고 정한다(짐작 금지).

### 현재 상태 — 보류(deferred), 폐기 아님 (★ 2026-06)

- **기술 가능성**: 사무실 PC(Windows)에서 Docker Desktop(WSL2)로 구현 가능. 블랙박스 계약은 OS 독립. 16GB는 e2-small(2GB)보다 여유. “Windows라서 불가”는 거짓.
- **보류 사유**: 도커 값은 H4(도커 vs AI리뷰) + ④ 되돌림 루프(judge_error·check_consistency)가 켜질 때만 발화. 그 둘은 v0 밖(9.5장), 미가동. 현재 연구는 H1→Q8(실행채널)로 흘렀고, Q8 발견은 도커 아닌 verify_channel(stdout 재생 실측)이 잡음.
- **현재 대체**: runner.py가 subprocess 직접 실행(timeout 20s), 관측은 verify_channel. `skip_docker=True` 기본 유지.
- **재발화 조건(하나라도)**: ④ AI검증 루프를 켤 때 / subprocess 직접실행에서 호스트 오염·위험동작 실측 시 / 생성물이 requirements 설치를 요해 격리 필요 시.

## 8. 검증·판정

> ★ 아래 “26B”(AI 판정자)는 현재 gemini-3.5-flash(3장). 코더와 동일 모델이라 담합방지 전제 깨짐 — 이 장 끝 “담합방지 전제 붕괴” 참조.

③ 정합성 검토(4장)를 계단식 파이프라인으로 확장한 장. 원칙 하나로 관통한다: **판정을 완벽 자동화하려 하지 않는다. 결정론적으로 판정 가능한 건 싼 기계 도구에 맡기고, AI(26B)는 도구가 못 하는 것만, 최종 본질 판단은 사람(9장 2층)이.** 이 순서가 한도(RPD 1,450) 절약과도 맞는다 — 명백한 에러를 26B에 보내는 건 낭비.

### 계단식 검증 파이프라인 (싼 것부터, AI는 맨 끝)

도커 실측(④) 전후로 아래 순서로 거른다. **위 단계에서 걸리면 아래로 안 내려간다**(AI 호출·한도 절약).

```
[0] Rule Engine   실행/도커 로그를 규칙(정규식 등)으로 1차 분석. 0원·즉시.
                  에러 종류로 design/code/environment 1차 태깅(에러 3계층, 아래).
[1] 정적 검사      ruff(문법·미정의 변수)·mypy(타입·시그니처 정합성). 코드를 실행 안 하고
                  검사. 쌈. 정합성 문제 상당수가 여기서 잡힘.
[2] 테스트 실행    pytest를 도커 안에서. "기능을 실제로 하는가"가 여기서 실측됨(바닥선 테스트).
[3] AI 판정(26B)   위 [0~2]가 못 잡은 것만. 비쌈 → 맨 마지막, 최소한만.
```

- **각 단계 출력은 공통 Issue 형식**(verdict/target/title/detail — 4장, 2부 지시서 4)으로 통일. 5번 루프는 한 형식만 보고 라우팅한다.
- ruff·mypy·pytest는 표준 도구라 새로 만들 게 없다 — **설치해서 호출만** 한다. ruff·mypy는 0원·0.1초급이라, 이걸 26B로 대신하면 한도를 태우면서 더 부정확하다.
- **함정**: pytest는 정확하지만 그 *테스트의 질*은 26B가 만든 것에 달려 있다. “에러 없이 실행됨 ≠ 기능을 진짜 함”(7장 looks_alive·9장 1층 함정과 동일). 그래서 [1]·[2]만으론 부족하고 아래 바닥선 테스트가 합쳐져야 막힌다.

### 바닥선 테스트 (“껍데기” 거르기)

aaa의 테스트는 “잘 만들어졌나”를 판정하지 않는다. **“껍데기는 아닌가”**만 거른다(품질 판단은 사람 = 9장 2층). 완벽한 검증이 아니라 **바닥선**만 본다 — 체스면 “말을 움직이면 보드가 실제로 바뀌나”, BMI면 “키·몸무게 넣으면 숫자가 나오긴 하나”. 이 정도는 26B도 허술하게 만들기 어렵다.

- **합격**: 테스트가 실제 기능 함수를 호출하고 그 반환값을 **assert** 하는가.
- **불합격**: “import 되나 / 에러 안 나나”만 보는 껍데기 테스트 → 다시 생성.
- **생성 주체**: 평소엔 26B가 설계 단계에서 바닥선 테스트도 같이 생성(기본·자동). 중요한 주제일 때만 사람이 “이건 꼭 이렇게 돼야 해” 한 줄을 덧붙여 반영(옵션).
- **바닥선 높이는 한 번에 안 정한다.** 출발점은 “기능 함수 호출 + 반환값 assert”. 너무 허술히 통과되면 올리고, 멀쩡한 것도 떨어지면 낮춘다(조정은 13장 트리거 A).

### 에러 3계층 분류 & 라우팅

되돌림 verdict를 `design`/`code` 둘에서 **`environment`를 더해 셋**으로 확장한다. 이유: 환경 에러는 코드를 고쳐도 안 풀린다 — `ModuleNotFoundError`를 code로 보내면 영원히 못 고친다.

|에러 패턴(예)                                           |계층         |보내는 곳                        |
|---------------------------------------------------|-----------|-----------------------------|
|`SyntaxError`, `IndentationError`                  |code       |코딩(②)                        |
|`ImportError: cannot import name X`(A가 부른 함수 B가 없음)|design     |설계(①) — 정합성                  |
|`ModuleNotFoundError`, 라이브러리 설치 실패                 |environment|requirements/도커 설정           |
|`AssertionError`(바닥선 테스트 실패)                       |code       |코딩(②) — 로직                   |
|멀쩡히 돌다 중간에 터짐 / 애매                                 |?          |26B 판정 → 그래도 애매하면 **code 먼저**|

- 명백한 패턴은 **Rule Engine([0])이 자동 분류** → 26B 안 거침(환각 0·한도 0). 애매한 것만 26B로.
- 폴백은 “애매하면 code 먼저”(4장) — 설계 탓하기 전에 구현부터 의심하는 게 싸다.
- 위 패턴표는 **출발점**이다. 실제 패턴·judge_error 세부 규칙은 도커로 진짜 에러를 받아본 뒤 채운다(13장 트리거 A). 양쪽-분기(design·code 수정본을 둘 다 만들어 도커로 가려내기)는 애매 케이스 한정 옵션으로 보류(13장 트리거 B).

### known_failures — 같은 벽에 머리 박기 방지 (오답 노트)

에러 기록을 두 종류로 **분리**한다. 같은 모델이 같은 실패를 반복하는 걸 *횟수*가 아니라 *내용*으로 막는 장치다.

- **Failure Log** = “무엇이 났나”(에러 로그 그 자체, 단순 누적). LLM에 로그만 던지면 “이 에러의 가장 흔한 1차원적 해결책”을 다시 뱉을 확률이 높다.
- **known_failures** = “무엇을 하지 마라”(시도했다 실패한 접근의 목록). 예: “requirements 수정 방식은 시도했으나 실패 → **그 방향 재시도 금지**.” 같은 태스크 안의 dead_end(아래)를 다음 루프 프롬프트로 넘기는 통로다.

known_failures가 강한 이유는 **부정 제약(Negative Constraint)**이기 때문이다 — LLM을 수렴시키는 가장 강력한 프롬프팅이 “하지 마라”다. 안전장치 표의 “같은 이슈 3회 → 설계 롤백”이 *횟수*로 무한루프를 막는다면, known_failures는 1회차 실패를 다음 프롬프트에 부정 제약으로 박아 *내용*으로 사전 차단한다 — 더 빨리 수렴한다. 둘은 경쟁이 아니라 보완.

- **누적 dead_end를 되돌림 feedback에 함께 주입**한다. make_design/code_one_file이 Issue(이번 지적)뿐 아니라 **지금까지의 dead_end 리스트(known_failures)**도 프롬프트 상단에 받아, “이미 막힌 길”을 피해 생성한다(2부 review.py 시그니처·인터페이스 정합성 표에 반영).
- **dead_end는 현재 태스크 안에서만 유효하다.** 루프가 끝나면(성공이든 max_loops 중단이든) 폐기한다 — max_loops=5라 5회 안에서만 쌓이고 버려진다. **만료 카운터·점수화는 두지 않는다**: 5회 한정이라 거의 발화하지 않을 장치를 더하는 건 단순함 원칙(14장)에 어긋난다. 태스크를 넘어 쌓을지(영구화)는 P3로 미룬다(13장).

### MEMORY UPDATE는 2단 — 별도 요약 AI를 두지 않는다

dead_end를 뽑는 일(로그→“하지 마라”로 압축)을 또 하나의 LLM 요약 에이전트로 만들면 8장 핵심 원칙(“판정은 기계, AI는 최소한”)에 구멍이 난다. 게다가 요약이 틀리면 *틀린 오답 노트*를 양산한다. 그래서 2단으로:

- **[1단] Rule Engine([0])이 결정론적으로 우선.** 명백한 패턴(`ModuleNotFoundError: X` → “X를 requirements에 안 넣고 import한 방식 실패”)은 정규식으로 dead_end를 뽑는다 — 환각 0·0원.
- **[2단] 애매한 것만 26B가, judge_error와 같은 호출에서.** 별도 요약 모델을 부르지 않고, 어차피 원인을 판정하는 judge_error가 **같은 호출**에서 `dead_end`(실패한 접근)와 `next_guideline`(다음 코딩 모델에 줄 지침)까지 함께 반환한다. 호출 1번으로 “원인 + 다음 지침”을 받아 RPD도 안 새고, 판정과 지침이 같은 근거에서 나와 일관된다.
- Flash-Lite를 요약에 쓰는 건 부적합 — 요약은 “추론”인데 현재 thinking 버그로 신뢰 불가다(13장). confidence 점수 같은 “검증·재현 불가능한 숫자”도 두지 않는다(환각 위험).

### 안전장치 (안쪽 루프)

|장치                             |막는 것       |비고                                                                                                                                    |
|-------------------------------|-----------|--------------------------------------------------------------------------------------------------------------------------------------|

- 안쪽 자동 수정이 캡에 걸리면 **에러 로그와 함께 그대로 멈춰** 바깥 반복(사람)으로 넘긴다 — 멈춤은 디스코드 알림으로 통지(10장).

### TaskState — 루프 전체가 읽고 쓰는 단일 상태 객체

위 “상태를 시스템이 보유”를 실제로 담는 그릇. 안쪽 루프가 도는 동안 **지금 어느 단계·몇 번째 시도·무엇이 실패했나**를 한 객체에 모은다. AI 머릿속(대화 히스토리)이 아니라 이 객체가 진실의 출처라, 재접속·재시작에도 안 잃는다(2장 아이폰 SSH·3장 thinking 비누적과 같은 정신).

```
TaskState:
  task_id       : 태스크 식별자
  expected_type : 의도한 결합도 A~E (planner엔 숨김, 리스크 레지스터) — runs.jsonl용(9.5장)
  designed_files: planner가 설계한 파일명들 (H1a 원재료) — runs.jsonl용(9.5장)
  generated_files: coder가 실제 만든 {파일명:코드} (H1b 원재료) — runs.jsonl용(9.5장)
  stage         : 지금 단계 (design / coding / review / docker / output)
  attempt       : 같은 단계 재시도 횟수 (3회 롤백·max_loops 판정의 근거)
  status        : running / stopped / done / failed
  failures      : Failure Log — 시도별 에러 누적 [{attempt, stage, error}]
  dead_end      : known_failures — 시도했다 실패한 접근(다음 프롬프트에 부정 제약으로 주입, 태스크 1회 한정)
```

- **하는 일**: 루프 관리(어디까지 왔나)·종료 조건(attempt가 max_loops 도달?)·실패 추적(failures/dead_end)·디스코드 알림(멈춘 단계·이유를 stage/status에서 뽑아 통지 — 10장)·**루프 종료 시 결과 한 줄을 `runs.jsonl`에 떨궈 Failure Dataset에 적재(9.5장)**.
- max_loops 중단·롤백 카운트가 전부 이 객체의 `attempt`·`status`를 보고 갈린다. 루프(2부 `loop.py`)는 매 단계 TaskState를 갱신·저장하고, 되돌림 때 dead_end를 feedback에 실어 보낸다.

### 근거 (구조로 신뢰성 확보)

이 장의 구조는 다중 에이전트 실패 모드(MAST, 1,642 트레이스)에서 빈도 높은 것들을 직접 겨냥한다: 단계 반복(FM-1.3, 전체 1위 15.7% → 3회 롤백), 히스토리 손실(FM-1.4 → JSON 상태), 종료조건 미인식(FM-1.5, 12.4% → 하드 캡·`max_output_tokens` 항상 명시), 추론-행동 불일치(FM-2.6, 13.2%)·검증 없음/부실(FM-3.2 → 실측 기반 계단식 검증 + 바닥선 테스트), 잘못된 검증(FM-3.3, 9.1% → 검증을 AI가 아닌 기계 도구에 위임), 작업명세 위반(FM-1.1 → 바닥선 테스트가 실측으로 잡음). **★ 담합방지 전제 붕괴(2026-06)**: 정본 최초엔 코더(31B)·검증자(26B)를 다른 모델로 분리해 자기 코드 봐주기(collusive validation, FM-3.3)를 막았으나, 모델이 gemini-3.5-flash로 통일되며(3장) 이 분리는 무효. 위험은 현재 잠복(① AI검증 check_consistency·judge_error가 v0 밖이라 무대 없음, ② 현재 검증은 기계 관측 verify_channel). **재발화 조건(AI검증 켤 때)**: (a) 검증 역할만 다른 model로 분리(call_model이 model 받음, H2 재발화와 동일) 또는 (b) 검증을 끝까지 기계 도구에 위임해 AI 자가검증 회피. 상위 원칙(“판정은 기계, AI는 최소한”)은 모델 통일과 무관하게 유효. 핵심 교훈은 “프롬프트 손질 < 구조 변경”이고, 이 장 전체가 그 적용이다.
*근거 출처: MAST(1,642 트레이스, NeurIPS 2025) 14개 실패 모드. 바닥선 테스트가 겨냥하는 “컴파일은 되나 기능은 안 됨”은 ChatDev 체스 사례(규칙 검증 누락→런타임 버그), 정체 방지의 “개선된 것만 채택”은 ReLook의 Forced Optimization에서.*

## 9. 가능성 판단 — 2층 4항목

이 프로젝트의 1차 목표는 “완성된 코드공장 만들기”가 아니라 **“이 두 모델로 멀티파일+도커가 되는지 확인”**이다. 새 오븐을 쿠키 몇 판 구워 직접 먹어보는 시식 단계 — 단, **매번 같은 채점표로** 먹어야 “3판째부터 가운데가 안 익네” 같은 패턴이 보인다.

### 두 층

```
1층 (자동, 최소선): 도커에서 에러 없이 의도대로 실행된다 (시스템이 객관 확인 = 실측)
                   ※ "에러 없이 종료 ≠ 정상 동작" 함정(빈 프로그램·삼킨 예외·입력 대기 후
                     즉시 종료)은 looks_alive의 기대출력 확인으로 보강(7장).
2층 (수동, 본질):   내가 제시한 요구사항이 반영됐나 — 매번 같은 4개 항목으로 본다(아래).
```

1층은 시스템이 자동으로 거르는 최소 조건이고, 2층은 사람이 보는 본질이다. **“실행된다”는 최소 조건일 뿐** — 빈 프로그램도 실행은 된다. 진짜 판단은 “내가 시킨 걸 했나”다.

### 두 층을 반복으로 보면 — 안쪽 자동 / 바깥 사람

같은 두 층을 *반복(iteration)* 관점으로 보면 역할이 갈린다.

**안쪽 루프 (사람 개입 없음, 자동):** 설계→검토→코딩→검증→도커실측, 에러 나면 자동 수정→재실측을 반복해 **프로토타입 1개**를 끝까지 뽑는다. 도중에 사람에게 안 떠넘긴다. 안쪽의 목표는 “사람한테 넘길 만한가”가 아니라 **“이게 프로토타입 자격이 있나”** — 핵심 기능이 실제로 동작하는가(빈 껍데기·가짜 금지). 1층(자동 최소선)이 이 자격을 객관 확인한다.

**바깥 반복 (사람+클로드, 또는 새 정보):** 안쪽이 뱉은 프로토타입을 사람이 보고 “이 방향으로 키우자”→다음 버전→또 본다. 이건 시스템 안 자동 루프가 아니라, **프로토타입을 가운데 둔 사람과의 반복**이다. 2층(수동 4항목)이 여기서 작동한다.

이 구분이 중요한 이유 셋:

- **두 층의 성공 기준이 다르다.** 안쪽 = “프로토타입 자격(기능 동작)”, 바깥 = “더 낫게(사람 판단)”.
- **안쪽은 순차로 충분하다.** aaa의 목적은 병렬·속도가 아니라 *싸고, 사람보다 빠르게, 쌓아갈 토대*를 뽑는 것 → 안쪽 루프를 굳이 병렬로 안 돌려도 된다(한도 부담↓, 6장).
- **바깥은 새 정보 없이는 안 돈다.** 매 회차에 실측 에러 / 사람 피드백 / 프로토타입 전체를 본 재설계 중 하나라도 들어와야 좋아진다. 새 정보 없이 “더 잘해봐”만 반복하면 정체된다 → **자동으로 무한 반복하지 않는다**(8장 안전장치).

역할 분담이 깨끗해진다: **aaa의 일 = 0→1**(빈 화면 없애기, 돌아가는 뼈대, 완벽 불필요). **사람+클로드의 일 = 1→2→3**(방향·판단·보완). 그래서 aaa 안에 정교한 자동 평가·자동 개선을 욱여넣을 필요가 줄고, 시스템이 단순해진다.

### 2층 체크 항목 (점수표 아님 — “볼 곳을 고정”하는 용도)

1. **핵심 기능이 동작하나** — 시킨 일의 main 기능. (todo면 “추가하면 진짜 추가되나”)
1. **요구사항을 빠뜨렸나** — 3개 시켰는데 2개만 했나.
1. **시키지 않은 걸 멋대로 했나** — 엉뚱한 기능, 과한 추상화. (설계 JSON의 `assumptions`를 보면 26B가 멋대로 메운 가정이 바로 드러난다 — 5장)
1. **★ 손보는 데 얼마나 걸리겠나** — 가능성의 핵심 지표. “10분 고치면 됨” vs “처음부터 다시”.

4항목은 점수표가 아니라 **매번 같은 눈으로 보기 위한 체크리스트**다. 주제마다 보는 기준이 달라지면 “모델 한계가 여기구나” 감이 안 쌓인다. 4번(수정 소요)이 “손볼 수준이냐”를 재는 자 — 어차피 초안이니 **완벽한지가 아니라 고쳐 쓸 만한지**가 기준이다.

### 합격선은 “고쳐 쓸 만한가” 하나

- **가능성 단계는 합격선을 숫자로 못 박지 않는다.** 작은 주제 몇 개를 돌려 결과물을 **직접 보고** 감으로 잡는다.
- **목표 100%는 가능성 단계 기준이 아니다.** 100%(손볼 게 없음)는 눈높이(“손볼 게 많은 초안”)와 모순 → 가능성 단계엔 매번 불합격이 나와 오판 위험. 100%는 자동 코드공장(아래 위임 3단계의 끝)의 최종 목표로 미룬다.

### 판정 위임의 단계적 확장

가능성이 확인되면, 판정 권한을 사람 → AI로 **단계적으로** 넘긴다. 한 번에 넘기지 않는다.

```
[1단계] 사람이 판정          ← 지금. 소수 샘플을 직접 본다.
[2단계] AI 판정 + 사람 검증   ← 1단계가 만족스러우면. 26B에도 판정시키고
                              내 판정과 비교해 "AI가 믿을 만한가" 검증.
[3단계] AI 위주 + 가끔 감리   ← 2단계에서 AI가 믿을 만하면. 비로소 자동 코드공장.
```

핵심: **검증 없이 바로 위임하지 않는다.** AI 자가평가는 후하거나 헛짚기 쉬우니(이게 검증을 기계 도구에 맡기는 이유 — 8장), 내 판정과 나란히 비교해 신뢰도를 확인하고 점점 위임한다. 판정은 “참고 → 검증 → 판정”으로 권한을 단계적으로 올리는 스위치로 구현.

이때 “사람이 본다”의 트리거는 **디스코드 알림**이 당긴다 — 결과물 완성·루프 중단을 폰으로 받아, 사람이 SSH에 상주하지 않아도 “불러주면 가서 보는” 방식이 된다(10장).

## 9.5 시스템의 정체성 — 무엇을 만드는 장비인가

> 이 장은 새 기능이 아니라 **이미 문서 곳곳이 암묵적으로 말하던 것을 한 줄로 못 박는** 선언이다. 9장(1차 목표는 완성이 아니라 가능성 확인)·리스크 레지스터(H1~H4 측정)·11장(usage 추적)·8장(known_failures)이 전부 이 한 줄 아래로 정렬된다. 그동안 이 선언이 없어 독자가 매번 “이게 앱 공장인가 실험 장비인가”를 스스로 재구성해야 했다. 12장 동결(freeze)이 “지금은 측정만이 답”이라고 한 것의 정체성 버전이다.

### 본 시스템의 실제 1차 목표

**aaa의 1차 목표는 완성된 애플리케이션 생성기가 아니다.** 무료 LLM(26B/31B)·제한된 컨텍스트·저사양 인프라(e2-small)라는 조건에서, **멀티파일 코드 생성이 어디서 성공하고 어디서 무너지는지를 실측 데이터로 수집하는 것**이 우선 목표다. 따라서 이 시스템의 첫 번째 산출물은 *앱*이 아니라 **실패 데이터셋(Failure Dataset)**이다. 앱 생성 성공률을 끌어올리는 일은 실패 패턴을 분석한 *이후*의 2차 목표다.

이 한 줄이 바꾸는 것: 정체성이 “작은 앱 자동 생성기”에서 **“멀티파일 LLM 실패 연구 장비”**로 명시된다. 그러면 9장의 “100%는 가능성 단계 기준 아님”, 리스크 레지스터의 “성공률이 아니라 어디서 무너지나”, 13장의 “돌려봐야 보인다”가 전부 같은 목적을 가리키는 문장으로 읽힌다 — 빈 화면을 없애는 게(0→1) aaa의 일이고, 그 과정에서 나오는 *실패의 분포*가 진짜 수확물이다.

### ★ v0 핵심 원칙 — 관측이 먼저, 계산은 사람이 (이 장의 헌법)

> 12장 freeze가 “설계 추정을 멈추라”라면, 이 박스는 **첫 구현(v0)이 무엇을 해도 되고 무엇을 하면 안 되는지**를 못 박는다. 아래 네 줄이 이후 모든 장(H2 review_design·8장 known_failures·정적 분석 계층)의 *발화 시점*을 결정한다 — 그 장들은 v0가 아니라, v0가 뱉은 데이터가 그것을 호명할 때 켜진다.

```
원칙: v0는 자동 개선 시스템이 아니다. 자동 평가 시스템도 아니다.
      v0의 목적은 실패 데이터를 수집하는 것이다.
따라서:
  1. 판단보다 관측을 우선한다.           (judge가 아니라 thermometer)
  2. 계산된 지표보다 원시 사실을 저장한다. (비율·버킷은 사람이 사후에)
  3. 계측기는 실측 데이터가 요구할 때만 추가한다. (선제 설치 금지)
  4. AI 판정은 v0 범위 밖이다.            (해석하는 AI는 v1 이후)
```

이 네 줄 중 **2번(관측 사실 ≠ 계산된 값)이 이 장 전체의 급소**다. 이번 설계에서 반복해 등장한 유혹 — `contract_accuracy`, 성공/실패/관측불가 **버킷**, `planner_failure_rate`·`coder_failure_rate` — 은 전부 *계산 결과*다. 계산 결과를 로그에 박으면 두 가지가 동시에 깨진다. ① 분모를 누가 쥐었는지에 따라 숫자가 뒤집힌다(예: planner가 계약을 적게 설계하면 분모가 줄어 “정확도”가 거짓으로 오른다 — C를 A로 다운그레이드한 모델이 더 높은 점수를 받는다). ② “구현됐다/실패다”의 판정 기준이 코드에 굳어, 30행을 돌려보기도 전에 *안 본 데이터로 채점 규칙을 도박*하게 된다(freeze 위배). 반면 `designed_files`·`generated_files`·`stderr`·`exit_code` 같은 **관측 사실**은 누가 봐도 같은 값이고, 분류·집계는 데이터가 쌓인 뒤 사람이 손으로(또는 굳은 규칙으로) 한다. **로그는 사실만 적고, 비율과 버킷은 계산하지 않는다.** 이 한 줄이 박혀 있으면, 이후에 `contract_accuracy` 같은 매력적인 지표가 다시 떠올라도 스스로 방어된다.

3번(계측기는 호명될 때만)의 구체는 아래 runs.jsonl 소절의 “0단계 → 점진 추가”에, 1·4번(관측 vs 판단)의 구체는 8장 계단식 파이프라인에서 *판단·라우팅을 떼고 관측만 남긴* v0 observer로 이어진다.

### Telemetry First — 데이터부터, 분류기는 나중

문서에는 Failure Classifier·judge_error 세부 규칙·Rule Engine 패턴표·known_failures 영구화 같은 *미래 기능*이 많다. **지금 단계에선 그 전부가 추측이다.** 그래서 원칙을 못 박는다: **Failure Classifier·Retry Policy·Context Compression은 실측 데이터를 확보한 *뒤*에 설계한다.** 데이터 없이 분류기를 먼저 짜지 않는다 — 이건 14장 “짐작 금지, 자료로 확인”과 8장 “판정은 기계, AI는 최소한”을 데이터 수집 순서에 적용한 것이다.

이 원칙이 13장 트리거 A(“도커로 진짜 에러를 받아본 뒤”)와 겹쳐 보이지만 결이 다르다. 트리거 A는 *개별 규칙*(judge_error 세부·패턴표)을 언제 채울지이고, Telemetry First는 그 위의 *순서 원칙* — “어떤 분류·정책이든 그 설계는 데이터를 본 다음”이다. 트리거 A가 이 원칙의 첫 적용 사례다.

### Failure Dataset — `runs.jsonl` (지금 시작, SQLite는 나중)

“실패 데이터셋”의 실체는 거창한 이벤트 로그가 아니라 **태스크가 끝날 때마다 결과 한 줄을 append하는 jsonl 파일**이다(`runs.jsonl`). 8장 TaskState가 이미 들고 있는 값을 루프 종료 시점(성공·실패·max_loops 중단 가리지 않고)에 한 줄로 떨군다.

**스키마는 고정하지 않는다 — 0단계 최소로 시작해, 데이터가 계측기를 호명할 때 키를 늘린다.** 이게 jsonl을 고른 이유 그 자체다(스키마리스). 필드 목록을 처음부터 완성해 박으면, 우리가 2부에서 비판한 “H1·H2·H3를 한 번에 구현”을 *로그 설계에서 그대로 반복*하는 셈이다. 그래서 위 v0 원칙 3번(계측기는 요구될 때만)을 로그에도 적용한다.

```
[0단계] 첫 5~10회 — 진짜 최소. observer 없이, 사실만:
  task_id          태스크 식별자 (TaskState.task_id)
  expected_type    던질 때 의도한 결합도 A~E (planner엔 숨긴다 — 적응 방지)
  designed_files   planner가 설계한 파일 목록 (설계 JSON files의 name들)   ← H1a 원재료
  generated_files  coder가 실제 만든 파일 목록/내용                         ← H1b 원재료
  exit_code        runner 실행 결과 (기계 판정)
  stderr           실패 메시지 원문 (해석 안 함, 통째로)
  runtime          실행 소요(초)
  created_at       PT 기준 타임스탬프 (11장 날짜 경계와 동일)
```

- **designed_files ↔ generated_files 쌍이 0단계의 핵심**이다. observer(정적 도구) 없이도 이 둘의 대조만으로 가장 비싼 신호가 공짜로 나온다: `expected_type=C`로 planner가 `[model, service, storage, api]`를 설계했는데(H1a 성공) coder 결과가 `[main.py]` 하나면(H1b 실패) — 결합도가 *설계 단계*가 아니라 *구현 단계*에서 무너졌음이 한눈에 잡힌다. 어느 모델의 고장인지(planner냐 coder냐)가 갈려야 후속 기능(review_design이냐 check_consistency냐)이 데이터로 결정된다(아래 H1a/H1b/H1c 분리, 리스크 레지스터 H1).
- **0단계 로그를 사람이 직접 읽는다.** 그러다 반복 패턴이 보이면, *그 패턴을 설명하는 가장 싼 계측기 하나*를 붙인다(observer 전체가 아니라 하나씩). 예: stderr의 다수가 `SyntaxError`면 → `py_compile:{ok}` 한 키 추가(compile에서 죽었나 runtime에서 죽었나를 재구성 없이 즉시 가른다). compile은 통과하는데 runtime만 터지면 → `ruff` 추가. 정적으론 멀쩡한데 계약 불일치가 의심되면 → `mypy:{errors, annotation_coverage}` 추가. 각 계측기는 “기능”이 아니라 온도계다 — 해석하지 않고 사실만 출력한다.
- **★ 계측기를 붙여도 판단·라우팅·집계는 하지 않는다.** observer는 8장 계단식 파이프라인에서 *판단과 되돌림 라우팅을 떼고 관측만 남긴 것*이다. compile이 실패해도 멈추지 말고 runtime까지 다 돌려 전 계층 로그를 남긴다(“compile은 깨졌는데 runtime은 어땠을까”의 상관이 가장 값진 행인데, 멈추면 영영 안 보인다). 그리고 `mypy`를 붙이더라도 **design.json을 mypy에 먹이지 않는다** — schema(문자열 `{text:str,done:bool}`)를 타입으로 번역하는 순간 그 번역 규칙이 숨은 채점 기준이 되어 observer가 judge로 자란다. mypy는 코드 *내부*의 자기모순만 보게 두고, design 대조는 사람이 designed_files·generated_files(날것)로 한다.
- **`annotation_coverage`를 mypy_errors와 반드시 같이 적는다.** 무료 모델은 타입 힌트를 잘 안 써서, `mypy_errors:[]`가 “계약 유지”와 “mypy가 장님이라 볼 게 없었음”을 똑같이 표시한다(7장 “에러 없이 종료 ≠ 정상 동작”의 정적 버전 = “mypy 통과 ≠ 계약 유지”). coverage가 낮은데 errors가 비었으면 그건 성공이 아니라 *관측 불가*(H1c)다. 단 coverage 임계(몇 %부터 관측가능이냐)는 v0에 박지 않는다 — 3회 롤백의 “3”과 같은 함정이라, coverage를 날것 숫자로 남기고 임계는 데이터를 본 뒤 정한다.
- **버킷·비율은 로그에서 계산하지 않는다(v0 원칙 2번).** 성공/실패/**관측불가** 3분류도, `contract_accuracy`도, 단계별 failure %도 전부 *계산된 값*이라 위 사실 필드에서 사후에 사람이 만든다. 단 사후 분류에 한 가지 우선순위는 못 박아 둔다 — **실패 증거(compile_error·ruff_error·exit_code≠0·stderr 중 하나라도)가 있으면 coverage가 낮아도 “실패”이지 “관측불가”가 아니다.** 관측불가는 *모든 관측 도구가 침묵했는데 깨진 증거도 없을 때*만 성립한다. 이 우선순위가 없으면 “관측불가” 버킷이 실패 세탁장이 된다(증거 있는 고장이 “우리가 못 봤다”로 둔갑).
- **왜 SQLite가 아니라 jsonl인가**: 최초 20~30개 프로젝트에서 보려는 건 실패의 *원인*이 아니라 *분포*다(“어느 난이도 칸부터, 어느 단계에서 깨지기 시작하나”). 30행은 `cat`·`tail`로 눈에 다 들어오고, 한 줄이 한 태스크라 아이폰 SSH(2장)에서도 읽힌다. SQLite의 질의력은 “손으로 못 볼 규모 + 어떤 컬럼으로 자를지가 굳은” 시점에야 값을 하므로, 지금 테이블을 짜면 *안 본 데이터로 스키마를 도박*하는 셈이다(12장 freeze 위배). 무엇보다 jsonl은 스키마리스라, 막상 돌려보고 “이 필드가 더 중요하네”가 나오면 다음 줄부터 키 하나 더 적으면 끝이다.
- **승급 경로**: 분포만 보다가 “C단계 × ImportError × loop 3회 이상”처럼 *이리저리 잘라 질의*하는 일이 잦아지면, 그때 `runs.jsonl`을 그대로 SQLite로 붓는다(파이썬 열 줄). 그 시점엔 어떤 컬럼으로 질의하는지를 *겪어서* 알기에 테이블을 제대로 짠다. SQLite 도입은 freeze 풀린 뒤(13장 트리거 B 계열)로 미룬다 — RAM(2GB)이 막는 게 아니라 freeze가 막는 것이다(`sqlite3`는 표준 라이브러리라 인프라 부담 자체가 없다).
- 위 0단계 필드는 **데이터 모양**이니 지금 확정해 둔다(스키마가 *고정*이라는 뜻이 아니라, 0단계 출발점이 이거라는 뜻 — 계측기 키는 호명될 때 는다). jsonl로 흘리든 나중에 SQLite로 붓든 그대로 부으면 된다. 구현은 토대(11장 사용량 모듈 옆) — TaskState에 `expected_type`만 새로 추가하면 `designed_files`·`generated_files`는 설계 JSON·코딩 출력에서, 나머지는 기존 필드 매핑이다(2부 state.py). **`task_type`(=expected_type)은 planner 프롬프트에 넣지 않는다**(난이도를 알면 모델이 실험에 적응한다 — 리스크 레지스터 측정 틀).

> 이 섹션은 *기능 목록*이 아니라 **이미 믿고 설계에 깔았지만 아직 측정 안 한 가정**을 모은 곳이다. 13장(아직 안 정한 것)이 “만들지 말지 보류한 기능”이라면, 여기는 “만들 게 아니라 사실인지 재볼 것”이다. 그래서 13장에 넣지 않고 별도로 둔다 — 프로젝트가 커질수록 “우리가 무엇을 아직 모르는가”를 적어둔 이 목록이 기능 목록보다 더 중요해진다.

지금 문서 곳곳의 근거는 둘로 갈린다. MAST 트레이스·공식 이슈처럼 **외부에서 검증된 것**과, 아래 H1~H4처럼 **우리가 그럴듯하다고 믿는 것**. 후자가 aaa의 토대인데 정작 한 번도 측정되지 않았다. 이를 한곳에 모아 “이건 가정이다”라고 명토 박는 게 14장 “짐작 금지, 자료로 확인”의 정직한 연장이다.

**공통 측정 틀.** 각 가설은 동일 PRD(요구사항)를 **10회 이상 반복**해 측정한다. 지표는 네 가지 — **성공률**(9장 “고쳐 쓸 만한가” 판정 기준), **평균 수정 횟수**(9장 2층 4번 = 가능성의 핵심 지표), **평균 루프 수**(되돌림이 몇 바퀴 돌았나), **토큰 사용량**(11장 usage.json의 calls·in·out·sec이 그대로 데이터 출처). 비교군을 바꿔가며(JSON 있음/없음, 분업/단독 등) 같은 지표를 잰다.

**측정용 PRD는 난이도 계단으로 — 랜덤하게 넣지 않는다.** 위 “10회 반복”이 *몇 번 도나*라면, 무엇을 던지나는 아래 A~E 계단으로 고른다. 주제를 무작위로 뽑으면 “모델 한계가 어디”라는 감이 안 쌓인다(9장 “매번 같은 눈으로”의 PRD 버전). 핵심은 성공률 자체가 아니라 **어느 칸부터 무너지기 시작하나**를 보는 것 — 그게 9.5장 Failure Dataset이 잡으려는 분포다.

```
A. 단일 파일 (멀티파일 이전의 바닥선)
   계산기 · 단위 변환기 · CSV 처리기
B. 멀티파일 CRUD (파일 간 인터페이스 등장)
   Todo API · 메모 API · 사용자 CRUD
C. 상태 공유 (데이터 계약이 진짜 시험되는 곳 — 5장)
   가계부 · 재고 관리 · 은행 계좌 모델
D. 이벤트 체인 (한 동작이 여러 파일을 연쇄로 건드림)
   주문→재고 감소→알림 · 회원가입→포인트→이메일
E. 의도적 실패 영역 (무너지는 걸 보려고 일부러 어렵게)
   권한 시스템 · 쿠폰+재고+장바구니
```

- A는 멀티파일이 아니라 **바닥선**이다 — 여기서도 깨지면 모델 자체 문제고, 여기는 되는데 B부터 깨지면 “멀티파일 정합성”이 범인이라는 게 분리된다(3장 약점 가설과 연결).
- C·D가 5장 데이터 계약(input/output_schema)과 H1이 진짜 시험되는 칸이다. 상태 공유·이벤트 체인은 “이름은 맞는데 데이터 모양이 어긋나는” 함정이 가장 잘 터지는 곳.
- E는 통과를 기대하지 않는다. **어디서 어떻게 무너지는지를 수집하려고** 일부러 넣는다 — 9.5장 “첫 산출물은 실패 데이터셋”의 가장 노골적인 칸. 이 칸의 실패가 `runs.jsonl`에서 가장 값진 행이 된다.
- 이 A~E가 `runs.jsonl`의 `expected_type` 값이다(9.5장). 칸별로 묶어(사후 집계) “계단의 어느 단에서 기울기가 꺾이나”가 분포로 드러난다 — 단 칸별 성공률 같은 *비율*은 로그에 저장하지 않고 사람이 사후에 만든다(9.5장 v0 원칙 2번).

각 가설에 **상태·영향도·우선순위**를 단다. 영향도는 “거짓일 때 무엇이 흔들리나”다 — **H1·H4는 거짓이면 장(章)을 뒤집고**(5장 / 7장), H3는 거짓이면 기능 하나를 뗄 뿐이다. 같은 미검증이라도 무게가 달라, **영향도 높은 H1·H4부터** 측정한다(측정 가능 시점으로도 H1이 가장 빠르다 — 아래 “언제 잰다”).

### H1. Design JSON이 효과가 있는가 → 세 갈래로 분리(H1a/H1b/H1c)

원래 H1은 “설계 JSON이 효과 있는가”라는 한 문장이었으나, 이건 서로 다른 세 고장을 한 덩어리로 묶고 있었다. 5장이 “데이터 계약 불일치가 멀티파일 실패의 큰 몫”이라 적어놓고도 H1 정의는 뭉툭했던 탓이다. **H1의 진짜 측정 대상은 “멀티파일 생성”이 아니라 “데이터 계약을 파일 경계 너머로 유지하는가”(contract retention)**다 — `user.py→service.py→repository.py`를 지나며 `User` 구조가 안 깨지는가. 멀티파일 생성 자체는 쉽다(파일 셋 만들면 끝). 어려운 건 계약 전파다. 그래서 셋으로 가른다. 셋은 다른 종류의 문제이고, **다른 후속 기능이 듣는다.**

**H1a — planner가 의도한 결합도를 *설계*하는가**

- **상태**: 미검증 · **영향도 높음**(거짓이면 5장 컨텍스트 관리 전제 붕괴) · **우선순위 높음**
- **가설**: `expected_type=C`(상태 공유)를 주면 planner가 실제로 파일을 분리하고 파일 간 계약(provides의 input/output_schema)을 *설계*한다 — 한 파일에 다 욱여넣지 않는다.
- **관측(계산 아님)**: `expected_type` ↔ `designed_files`. C를 줬는데 designed_files가 1개면 planner가 C를 A로 다운그레이드한 것. observer 없이 잡힌다.
- **거짓이면**: review_design(H2) 후보가 여기서 정당화된다 — planner 단계에서 결합도를 강제·교정.

**H1b — coder가 *설계된* 결합도를 *유지*하는가**

- **상태**: 미검증 · **영향도 높음**(거짓이면 5장 데이터 계약 전제 붕괴) · **우선순위 높음**
- **가설**: planner가 분리 설계(designed_files 여럿)를 냈을 때, coder가 그 계약을 코드에서 유지한다 — main.py 하나로 뭉개지 않고, 파일 간 호출의 데이터 모양이 어긋나지 않는다.
- **관측**: `designed_files` ↔ `generated_files`(구조 유지됐나) + (계측기 붙은 뒤) `mypy` 계약 모순. C를 설계했는데 generated가 main.py 하나 = H1b 실패.
- **거짓이면**: check_consistency(③)가 여기서 정당화된다 — 코드 단계에서 계약 깨짐을 잡는다.

**H1c — 우리가 H1b를 *관측*할 수 있는가** (코드 문제가 아니라 *계측* 문제)

- **상태**: 미검증 · **영향도 높음**(거짓이면 H1b를 영영 사람 눈에만 의존 → 50회가 천장) · **우선순위 높음**(H1b와 한 몸)
- **가설**: 정적 도구(mypy 등)가 H1b의 계약 유지/파괴를 실제로 볼 수 있다.
- **관측**: `annotation_coverage`. coverage가 낮으면 mypy는 장님이고, `mypy_errors:[]`는 “계약 유지”가 아니라 “관측 불가”다(“mypy 통과 ≠ 계약 유지”, 7장 함정의 정적 버전).
- **왜 별도 가설인가**: 관측불가를 H1b 실패로 세면 거짓말이 된다. 코드가 나쁜 게 아니라 *우리가 못 본* 상태다. 이걸 분리해야 “성공 42 / 실패 38 / **관측불가 20**”이라는 정직한 결과가 나온다 — 관측불가 버킷은 시스템을 바꾸지 않고 현실을 드러낸다. 타입 힌트를 *강제*해 coverage를 올리는 건 H1을 다른 실험(“설계 JSON + 강한 타입 규율”)으로 바꾸고 소형 모델에선 로직을 형식 코드로 밀어낼 위험이 있어, v0에선 coder에 “가능한 경우 타입 힌트 사용”만 권장하고 강제하지 않는다. coverage는 그저 관측한다 — coverage↔성공률 상관이 *데이터로* 보이면, 그때 비로소 “타입 강제”를 실험 변수로 올린다.
- **공통 비교군**: 위 셋 모두 “설계 JSON 경유 vs JSON 없이 직접 생성”을 같은 PRD 10회씩으로 잰다. 분리의 효용은, 직접 생성이 더 나아도 *어느 단계*에서 JSON이 지거나 이기는지(설계냐 구현이냐 관측이냐)가 갈린다는 것.
- **건드리는 것**: 5장 전체(시그니처·스키마 공유)의 존재 이유. H1b·H1c가 거짓이면 “input/output_schema만으로 계약을 유지한다”는 1부의 핵심 전제가 흔들린다. 그리고 이 분리가 **후속 기능의 발화 순서를 데이터로 정한다** — 실패가 H1a에 몰리면 review_design을, H1b에 몰리면 check_consistency를 먼저 켠다(둘을 동시에 켜면 어느 쪽이 효과인지 분리 불가).

### H2. 26B 설계 + 31B 구현 분업이 효과가 있는가

- **상태**: 미검증 · **영향도 중간**(거짓이면 3장 역할 배치 재고, 단 단일 모델로도 굴러감) · **우선순위 중간**
- **가설**: 26B가 설계하고 31B가 구현하는 분업이, 31B 단독(설계+구현 모두)보다 효율적이다(성공률↑ 또는 루프 수↓).
- **비교**: 26B/31B 분업 vs 31B 단독 vs 26B 단독. 같은 PRD 10회씩.
- **건드리는 것**: 3장 역할 분리. 담합 방지(8장 collusive validation) 근거이기도 해서, 분업 이득이 없으면 역할 배치를 재고.

### H3. known_failures가 수렴성을 높이는가

- **상태**: 미검증 · **영향도 낮음**(거짓이면 known_failures 기능만 제거, 다른 장 영향 없음) · **우선순위 낮음**
- **가설**: 실패한 접근을 다음 프롬프트에 부정 제약으로 주입하면(8장), 주입 안 할 때보다 더 적은 루프로 수렴한다.
- **비교**: known_failures 주입 on vs off. 같은 PRD에서 일부러 실패를 유도해 10회씩.
- **건드리는 것**: 8장 known_failures와 P3 영구화(13장) 판단의 근거. 효과가 확인돼야 영구화를 논할 가치가 생긴다.

### H4. Docker 검증이 AI 리뷰보다 효과가 있는가

- **상태**: 미검증 · **영향도 높음**(거짓이면 7장 “실측이 추측을 이긴다” 철학이 흔들림) · **우선순위 높음**
- **가설**: 도커 실측(④)이 26B의 코드 리뷰보다 “실제로 도는가”를 더 정확히 가른다(AI가 “문제없다”지만 도커에서 터지는 사례를 잡는다).
- **비교**: 도커 실측 결과 vs 같은 코드에 대한 26B 리뷰 판정. 불일치 건수·방향을 집계.
- **건드리는 것**: 4·7장의 “실측이 추측을 이긴다” 핵심 주장. aaa의 가장 큰 차별점이라 가장 먼저 확인할 가치가 크다.

**언제 잰다.** 코드를 새로 짜서 `skip_docker=True`로 ①~③+⑤가 돌기 시작하면 H1·H2·H3는 바로 측정 가능하다(도커 없이도 성공률·루프 수·토큰이 나온다). 특히 **H1a·H1b는 0단계 로그(`expected_type`·`designed_files`·`generated_files`)만으로 observer 없이 즉시** 갈린다(9.5장). **H1c는 mypy 계측기를 붙인 뒤**(0단계 로그가 그것을 호명할 때) `annotation_coverage`로 측정된다 — 즉 H1c의 측정 가능 시점 자체가 “데이터가 계측기를 부른 뒤”라, 이 가설은 자기 측정 도구의 도입 시점까지 데이터에 맡긴다. H4는 `run_in_docker` 본문을 채운 뒤(12장 다음 단계 3)에야 잴 수 있다. 즉 이 레지스터는 12장의 “결과물로 가능성을 본다”와 같은 시점에 채워지기 시작한다.

### 재설계 트리거 — 언제 이 구조를 버리나

대부분의 계획이 “계속 개선한다”만 있고 “언제 포기하나”가 없다. aaa도 무한히 가지를 치며 손보다 보면 *연구 장비가 연구 대상보다 무거워지는* 순간이 온다. 아래 조건 중 하나라도 만족하면 **기존 구조의 확장을 멈추고 재설계를 검토**한다 — 더 고치는 게 아니라 틀을 다시 본다.

```
1. max_loops 도달률이 너무 높다     → 되돌림이 수렴이 아니라 발버둥. 자동 루프의 전제가 깨짐
2. 사람 디버깅 시간 > AI 생성 시간   → "사람보다 빠르게"(9장)가 거짓이 됨. 장비가 짐이 됨
3. dead_end 크기 > task 크기         → 막힌 길 목록이 과제보다 큼. 8장 known_failures가 역효과
4. 감독 코드 규모 > 생성 코드 × N    → 검증·라우팅 인프라가 결과물을 압도. 단순함 원칙(14장) 붕괴
5. retry 성공률이 통계적으로 무의미   → 되돌림이 결과를 못 바꿈. H3·되돌림 전체의 근거 상실
```

- **숫자는 지금 박지 않는다(측정 후 확정).** 1의 “너무 높은”, 4의 “N배”, 5의 “무의미”는 전부 H1~H4를 재기도 전에 박으면 또 짐작이다(13장 “3회 롤백의 3도 실측 후”와 같은 함정). `runs.jsonl`이 20~30행 쌓이면 max_loops 도달률·평균 루프가 분포로 나오니, 그 실측을 보고 임계값을 채운다. 그래서 여기는 **트리거의 *틀*만** 두고 숫자는 빈칸으로 둔다.
- 이건 9장 위임 단계(사람→AI)와 반대 방향의 안전장치다. 위임이 “잘 되면 권한을 넘기는” 스위치라면, 재설계 트리거는 “안 되면 틀을 버리는” 스위치다. 둘 다 같은 `runs.jsonl` 데이터를 읽는다 — 한쪽은 승급 신호로, 한쪽은 폐기 신호로.
- 이 트리거가 당겨지면 그 자체가 가장 값진 실패 데이터다(9.5장). “이 두 모델 + 이 구조로는 여기가 천장”이라는 결론은 앱 하나 성공보다 연구 장비로서 더 큰 수확이다.

## 10. 사람 개입 채널 (디스코드 알림)

자동 루프가 “사람이 본다” 지점에서 멈추거나 결과물을 뱉을 때, 사람이 SSH에 상주하지 않아도 폰으로 알도록 디스코드로 통지한다. 관리가 아이폰뿐(2장)이라, “능동적으로 들여다보기”를 **“불러주면 가서 보기”**로 바꾸는 장치. 2층 반복(9장)의 바깥 루프가 실제로 도는 진입점이 이 알림이다 — 알림이 없으면 사람이 언제 볼지를 몰라 바깥 반복이 안 돈다.

### 알림의 목적은 로그 전달이 아니라 판단 전달

알림이 하려는 일은 **에러 로그를 폰에 옮기는 게 아니다.** 운영자가 아이폰에서 한눈에 내릴 판단은 단 하나 — **“지금 SSH 켜고 가서 볼 가치가 있나(재시도 가치)”**다. 그래서 알림 본문은 로그가 아니라 *그 판단에 필요한 최소 정보*를 담는다. 전체 로그는 폰에서 읽을 것이 아니라 SSH에서 볼 것이므로 **첨부/파일로 저장**하고, 디스코드 본문은 **최대 20줄**을 넘기지 않는다(폰 화면에서 스크롤 없이 판단되게).

### 통지 지점

- 프로토타입 완성 / 루프 정상 종료 → “결과 나옴, 보러 와” (9장)
- max_loops 초과로 중단 → “고쳐 쓸 만한지 사람이 판단” (출력 루프)
- RPD 안전선 임박·초과로 멈춤 → “한도, 멈췄음” (6장)
- (선택) 안쪽 루프가 같은 이슈 반복 등 에러 누적

### 통지 메시지에 담을 것 (판단용 최소 정보)

폰에서 “지금 가서 볼지”를 판단하려면 한 화면으로 맥락이 와야 한다. 9.5장 `runs.jsonl`이 들고 있는 값을 거의 그대로 쓰되, 거기에 *판단을 거드는 두 줄*을 더한다.

```
본문에 넣는 것 (20줄 이내):
  주제            요구사항 한 줄 요약 ([todo 앱] 등)
  결과            성공/실패 (exit_code 등 0단계 사실에서 사람·규칙이 읽어 표시 — 로그엔 사실만 저장, 9.5장)
  loop_count      되돌림 몇 바퀴 돌았나 (TaskState.attempt)
  failed_stage    어디서 멈췄나 (design/coding/review/docker — TaskState.stage)
  retry_value     재시도 가치 — "낮음(같은 에러 3회 핑퐁)" / "높음(새 에러, 한 번 더 가볼 만)" 류
  recommended_action  추천 행동 — "사람 판단 필요" / "요구사항 다듬어 재투입" / "그냥 폐기" 류
전체 로그       → 첨부/파일(SSH에서 본다). 본문에 붙이지 않는다.
```

- `retry_value`·`recommended_action`은 14장 “선택지만 나열 말고 추천을 같이”의 알림 버전이다 — 방향 결정을 폰 든 사람에게 떠넘기지 않고, 시스템이 *판단 근거 + 추천*까지 한 줄로 올린다. 예: “[todo 앱] fail · loop 3 · review · retry_value 낮음(같은 ImportError 반복) · 추천: 사람 판단(설계 롤백 후보).”
- 단 **최종 결정은 사람**이다(14장). 알림은 “부르는 + 추천하는”까지고, 코드를 보고 채택할지는 SSH에서 사람이 한다.
- `retry_value`를 무엇으로 판정하나는 8장 안전장치(같은 이슈 3회·dead_end 크기)에서 이미 나오는 신호를 재사용한다 — 새 판정 호출(26B)을 만들지 않는다(한도·단순함). 규칙으로 못 가르는 애매한 경우만 “사람 판단 필요”로 올린다.

### 방식: 웹훅 (봇 아님)

- 알림(시스템→사람) 한 방향이라 웹훅 URL에 POST만. 상주 봇 불필요.
  → 라이브러리 없이 `requests`로 충분. systemd 서비스 추가 안 해도 됨.
- 폰 푸시가 실제로 뜨는지는 구현 때 1회 실측(짐작 금지).

### 보안

- 웹훅 URL은 키처럼 취급 — 코드/커밋 금지, 환경변수 `DISCORD_WEBHOOK_URL`, .gitignore(14장 보안 소절).

### 범위 (지금 / 나중)

- 지금: 알림(방향 1)만.
- 나중: 사람→시스템 명령(방향 2, “다음 버전 ㄱㄱ” 등)은 상주 봇·입력 파싱·발신자 검증이 딸려 가능성 단계엔 과함 → 위임 2~3단계로 자동화가 올라갈 때 재검토(13장 트리거 B).

## 11. 사용량 추적 + 비용 시뮬

> ★ 현재 단일 모델 gemini-3.5-flash(3장)라 모델별 누적도 사실상 1개. 아래 예시·서술의 26B/31B/gemma는 archive(Gemma 시절). 집계 위치는 3장 “사용량 집계 위치” 참조.

### usage.json — 지금부터 쌓는다

모델 호출 1건마다 **calls·in·out·sec** 네 가지를 PT 날짜별·모델별로 누적한다. 비용 시뮬은 나중에 붙이지만(아래), **데이터는 지금부터 쌓아야** 나중에 단가표만 곱하면 된다.

```
구조 예:
{
  "2026-06-01": {
    "gemini-3.5-flash": {"calls": 20, "in": 52000, "out": 33000, "sec": 243.9}
  },
  ...
}
```

- **in·out 토큰**은 응답의 `usage_metadata`에서(`prompt_token_count`, `candidates_token_count`).
- **sec(초)**는 호출 시작~끝(`time.time()`). 이게 쌓이면 모델별 평균 응답시간이 자연히 나온다 — 비용 시뮬과 속도 비교의 재료.
- **★ 날짜 키는 PT(태평양시) 기준.** RPD 리셋이 PT 자정이라(6장), 로컬시간으로 날짜를 끊으면 카운터가 RPD와 어긋난다. usage 누적과 RPD 체크가 같은 날짜 경계를 봐야 한다.

(구현은 토대의 사용량 모듈. 한도 슬롯이 RPD 체크에 오늘 누적 호출수를 쓴다.)

### 비용 시뮬 — 나중에, 단 테이블은 처음부터 분리

비용 시뮬은 “일단 만들고 넣자”로 미룬다. 데이터(위 calls/in/out/sec)는 쌓이니, 나중에 단가표만 곱하면 된다. 단 한 가지는 처음부터 지킨다:

- **단가/속도 테이블을 코드에 박지 말고 별도 파일(`pricing.py` 등)로 분리.** 모델을 갈아탈 때 테이블만 고쳐 “모델 X로 바꿨다면 얼마/몇 초?”를 시뮬레이션할 수 있게. 현재 26B/31B는 무료라 단가는 0이지만, 유료 모델 검토 시 이 분리가 그대로 레버가 된다.

### 속도는 예측 말고 실측 누적

응답 속도(t/s)는 **예측하지 않는다.** 제공자·시간대·부하에 따라 수배까지 출렁이기 때문이다(같은 31B라도 제공자 간 568% 차이 보고). 안 돌려본 모델의 절대 시간은 장담하지 않고, 위 `sec` 누적으로 **실측**해서 본다. — aaa의 “짐작 금지, 자료로 확인” 원칙이 그대로 적용되는 자리.

*(비용 시뮬 도입 시점·유료 모델 전환은 13장 참조.)*

## 12. 현재 상태

### 한 줄

종이 설계가 끝났고, 다음은 **새 설계 기준으로 코드를 처음부터** 짜는 단계다. (옛 코드는 폐기 — 이 문서가 정본.)

### 지나온 길

- **종이 설계**: 흐름·한도·컨텍스트·판정기준·검증 파이프라인·2층 반복까지 잡았다.
- **코드(폐기)**: 도커 전 단계(토대~출력)까지 한 번 구현해봤다. 단, 이 통합 작업으로 설계가 바뀌므로 **그 코드는 잇지 않고 처음부터 다시 짠다.** (옛 코드를 주워 이어붙이지 말 것 — 통합본이 정본)
- **도커**: 학습을 더 진행하지 않는다. 구현은 AI가 하므로 **개념만 알면 충분하고, 그 이해는 완료**됐다(7장 블랙박스 계약이 곧 구현 명세). 사람이 도커를 직접 다룰 게 아니라, 손 실습 전 과정을 밟을 필요는 없다.

### 지금 서 있는 자리

- **설계 단계 완료** — 흐름·한도·컨텍스트·검증·2층 반복까지 확정(이 문서). 코드는 아직 안 짬(다음 단계).
- **VM**: 만들어졌고 **도커도 설치됨**. 코드를 올려 돌릴 토대는 준비됨(배포 흐름은 2장).
- **가능성 판단(9장)**: 이번에 새로 정립된 개념 — 2층 4항목(핵심기능/누락/멋대로/수정소요)으로 결과물을 사람이 본다. 아직 실제 결과물로 판단해본 적은 없다(코드를 새로 짠 뒤가 그 첫 시점).

### 다음 단계 (순서)

1. **새 설계 기준으로 코드 구현** — 토대(모델 호출·한도 슬롯·사용량·TaskState)부터 의존성 순으로. 옛 코드 재사용 없음.
1. **`skip_docker=True`로 ①~③+⑤** 끝까지 돌려, 결과물을 2층 4항목으로 직접 본다(가능성 1차 감). 이때부터 H1·H2·H3를 측정하기 시작한다(리스크 레지스터, 9장 뒤).
1. **`run_in_docker` 본문 구현** → `skip_docker=False`로 도커 실측까지 연결(1층 자동 판정 작동). H4(도커 vs AI 리뷰)는 이 시점부터 측정 가능.
1. (실측 에러를 받으며) 검증 파이프라인·judge_error 세부 규칙·디스코드 알림을 붙인다.

### 설계 동결 (freeze) — H1 실측 전까지

종이 설계는 여기서 멈춘다. **지금 가장 위험한 건 문서가 아니라, 문서가 좋아졌다는 이유로 구현을 미루는 것이다.** 이 시점부터 설계를 더 토론해도 한계효용이 급락한다 — H1~H4는 더 깎아도 답이 안 나오고 측정만이 답이다. 게다가 측정 전에 “Design JSON이 왜 좋은지”를 더 정교하게 정당화할수록, 막상 H1이 거짓으로 나왔을 때 데이터를 받아들이기 어려워진다(반증에 둔감해진다). 그래서 지금 멈추는 게 인식론적으로도 옳다.

**동결의 대상은 ‘추정’이지 ‘사실’이 아니다.** 두 가지를 구분한다:

- **금지(동결)**: “이렇게 하면 더 좋을 것 같다”류의 새 아이디어·구조 변경·기능 추가. H1 결과가 나오기 전까지 전부 보류(13장에 적고 손대지 않는다).
- **허용(권장)**: 구현하다 부딪힌 *실측 사실*. 예 — `run_in_docker`를 subprocess로 짜보니 SDK가 단순하더라(7장), 26B 입력 한도가 키로 확인하니 128K더라(13장 빈칸), RPM 15가 실제론 더 빡빡하더라. 이건 “내 생각”이 아니라 데이터라 동결 대상이 아니다. 표·13장의 빈칸을 채우는 것뿐.

**행동 순서**(오너 관점, 단순하게): ① `state.py`(독립이라 첫 타자, save→load 단독 테스트) → `usage.py`·`limiter.py`·`client.py` → ② Docker Runner → ③ **H1 실험 10회** 실행 → ④ **결과를 이 문서에 적는다.** 그 시점부터 “내 생각”이 “우리 데이터”가 된다 — 14장 “짐작 금지, 자료로 확인”이 문서 적용을 넘어 프로젝트 운영 원칙으로 올라서는 순간이다.

## 13. 아직 안 정한 것

종이로 더 깎을 수 없어 미룬 것들. 공통 성격: **돌려봐야 보이거나, 지금 하면 과하다.** 무엇을 푸는지가 아니라 **언제·무엇으로 푸는지(트리거)**로 묶는다. 트리거가 오기 전엔 구현하지 않는다(섣불리 만들면 가능성 단계가 복잡해진다).

이 장은 “만들지 말지 보류한 **기능**”이다. “만들 게 아니라 사실인지 재볼 **가정**”(Design JSON·분업·known_failures·도커 우선이 정말 효과 있나)은 기능이 아니라 리스크라, 9장 뒤 “검증해야 할 가설(리스크 레지스터)”에 따로 두었다. 둘을 섞지 말 것.

### 트리거 A — 도커로 진짜 에러를 받아본 뒤

(실측 없이 규칙을 박으면 짐작이 된다)

- **judge_error 세부 규칙** — 어떤 실제 에러를 design/code/environment 중 어디로 보낼지. 큰 갈래는 잡혔고(8장 에러 3계층), 세부 패턴은 실측 후.
- **Rule Engine 패턴표** — 정규식 등으로 1차 분류할 실제 에러 패턴. 보강 메모의 표는 출발점일 뿐.
- **바닥선 테스트의 높이** — “기능 함수 호출 + 반환값 assert”로 출발해, 너무 허술히 통과되거나 멀쩡한 것도 탈락하는지 보며 조정.
- **3회 롤백의 “3”** — 같은 이슈를 몇 번 반복하면 설계로 롤백할지. 너무 잦거나 늦으면 조정.

### 트리거 B — 가능성이 확인된 뒤

(지금 넣으면 검증 안 된 토대에 얹는 셈)

- **acceptance_criteria 자동 테스트 생성 (P1)** — 지금은 설계 JSON에 필드만 둔다(5장). 이 criteria로 “통과하는 pytest를 26B가 짜게” 하고 바닥선 테스트를 강화하는 건 P1. criteria가 “테스트 생성 힌트”일 뿐 절대 기준이 아니라(빈 껍데기 테스트로 통과 가능), 자동화해도 최종 품질 판정은 사람이 유지한다(9장 2층).
- **비용 시뮬** — 데이터(calls·in·out·sec)는 지금부터 쌓되, 단가표 곱하기는 나중. 테이블 분리는 지금부터(11장).
- **판정 위임 스위치** — 사람→AI 위임의 단계 전환(9장 1→2→3단계). 1단계(사람)가 만족스러울 때 다음으로.
- **양쪽-분기를 애매 케이스에 실제로 쓸지** — design·code 수정본을 둘 다 만들어 도커로 가려내기. 호출·도커가 2배라 한도 여유를 보며. 안 쓰면 26B 판정 단독(8장).
- **여러 컨테이너 병렬 실행** — e2-small(2GB) 메모리 한도를 보며. 가능성 단계는 컨테이너 1개로 충분(7장).
- **Flash-Lite 잡일 분리** — `gemini-3.1-flash-lite`는 RPD 500(별도 쿼터)이라 메인 불가 + **현재 thinking 버그로 추론 신뢰 불가** → 무추론 잡일(JSON 검증·요약) 전용 후보로만 킵. 3단계에서 Gemma RPD가 빠듯해지면 재검토. 핵심 일(설계·코딩·판정)은 검증 안 된 모델에 안 맡긴다.
- **known_failures 영구화 (P3)** — dead_end를 지금은 태스크 1회 안에서만 쓴다(루프 끝나면 폐기). 태스크를 넘어 쌓으면(“Strategy Memory”) “이 두 모델은 늘 이런 실수를 한다” 패턴이 모여 강력하지만, 검증 안 된 토대에 영구 메모리를 얹는 셈이라 가능성 단계엔 과함. 영구화·만료·점수화 같은 확장은 전부 **P3**로 미룬다 — “known_failures가 수렴을 돕나”가 작은 주제 반복(9장)으로 확인된 뒤에 손댄다.
- **디스코드 방향 2 (사람→시스템 명령)** — “다음 버전 ㄱㄱ” 같은 역방향. 상주 봇·입력 파싱·발신자 검증이 딸려 가능성 단계엔 과함. 위임 2~3단계로 자동화가 오를 때(10장).
- **Failure Dataset SQLite 승급** — 지금은 `runs.jsonl` 한 줄 append로 분포만 본다(9.5장). “C단계 × ImportError × loop 3회”처럼 *잘라 질의*하는 일이 잦아지면 jsonl을 그대로 SQLite로 붓는다(파이썬 열 줄, 컬럼은 그때 겪어서 안다). RAM이 아니라 freeze가 막는 항목 — 분포로 부족해지는 게 확인된 뒤.
- **재설계 트리거 임계값 확정** — 리스크 레지스터 “재설계 트리거”의 다섯 조건은 *틀*만 잡혀 있고 숫자(max_loops 도달률 %·감독/생성 코드 N배·retry 유의성)는 비어 있다. `runs.jsonl`이 20~30행 쌓여 max_loops 도달률·평균 루프가 분포로 나온 뒤 그 실측으로 채운다(13장 트리거 A “3은 실측 후”와 같은 정신).

### 트리거 C — 결과물이 커지면

- **큰 파일 쪼개기** — 한 파일 출력이 32K를 넘으면 함수 단위로 나눠 여러 번 호출. 감지(`finish_reason == MAX_TOKENS` → `truncated`)는 지금 넣고, 쪼개기 구현은 닿을 때. 작은 주제엔 미발생 예상.

### 트리거 미정 — 도커와 함께 자리 잡을 것

- **②.5 단위검증 자리** — 회사식 “단위→통합” 순서에서 단위 자리. 지금은 통합(도커)만. 도커 본문과 함께 위치 확정.
- **26B 입력 한도 재확인** — 256K인지 128K(small)인지 내 키로 확인. 시그니처만 넣으면 실전 영향은 없지만 표의 빈칸.

# 2부 — 구현 명세 (어떻게)

> **2부의 위치.** 1부가 “왜/무엇”이라면 2부는 코드를 짤 때 보는 **함수 시그니처와 인터페이스 계약**이다. 구현은 의존성 순(토대→설계→코딩→검토→출력)으로 한 덩어리씩, 단독 테스트를 통과시키며 진행한다.
> 
> **공통 전제는 1부 참조.** 모델·호출문법·한도·세 함정은 3·6장, 답변 규칙·보안은 14장, 전체 흐름은 4장, 되돌림 Issue 형식은 4·8장, 설계 JSON 스키마는 5장. 2부는 이 전제 위에서 각 파일의 알맹이만 명세한다 — 모든 모델 호출은 토대의 `call_model(...)` 한 입구를 거치고, 되돌림은 전부 공통 Issue dict `{ok, issue:{verdict,target,title,detail}}` 하나로 흐른다.

class Limiter:

# state.py
class TaskState:
    """루프 전체가 읽고 쓰는 단일 상태 객체(8장). AI 머릿속이 아니라 이 객체가 진실의 출처 —

    dead_end:  list         # known_failures: 실패한 접근(다음 프롬프트에 부정 제약 주입, 태스크 1회 한정)
    def save(self) -> None: ...   # JSON으로 즉시 저장
    def add_failure(self, stage: str, error: str) -> None: ...   # failures 누적 + attempt 관리
    def add_dead_end(self, approach: str) -> None: ...           # dead_end 누적(중복 제거)

        # 루프 종료 시(성공·실패·중단 가리지 않고) 결과 한 줄을 runs.jsonl에 append — Failure Dataset(9.5장).
        # 0단계 필드(관측 사실만): task_id·expected_type·designed_files·generated_files·exit_code·
        #   stderr·runtime·created_at(PT). ★ 계산된 값(success/실패 버킷·비율) 금지 — 사람이 사후에(9.5장 v0 원칙).

**밖에 제공하는 것**: 2·3·4·5가 쓰는 단 하나의 입구는 `call_model(...)`. limiter/usage는 객체로 주입받을 뿐 직접 만지지 않는다. 반환의 `text`·`truncated`가 상위 단계의 판단 재료. `TaskState`는 `loop.py`가 만들어 각 단계에 넘기고, 단계들은 거기에 failures/dead_end를 쌓는다.

**단독 테스트**: 환경변수 `GOOGLE_API_KEY` 설정 후 ① limiter 하나로 `call_model`로 26B에 “1+1=?” 호출 → 답이 오나, usage.json에 calls=1/in/out/sec 찍히나. ② 연속 20번 호출해 RPM 15에서 16번째쯤부터 간격이 벌어지나. ③ TaskState 만들어 add_failure·add_dead_end 후 save→load 하면 그대로 복원되나(상태 비손실).

**주의점**: PT 날짜 경계 꼭 맞출 것(usage·RPD 둘 다). RPD 초과는 일단 **예외 던지고 멈춤**으로 시작(자정 대기는 자동 루프에서 위험 — 멈추고 사람이 본다). `max_output_tokens` 절대 빼지 말 것(무한 thinking 함정 — 3장). TaskState는 매 단계 변화 직후 save(재접속 대비) — 만료·점수화는 넣지 말 것(8장).

## 설계 (① + ①.5) — `design.py`

요구사항을 받아 **파일 구조 + 각 파일 역할 + 시그니처**를 JSON으로 확정하고(①, 26B), 코딩 전 **구현 가능·모순 없음**을 검문한다(①.5, 31B). 막히면 ①로 되돌린다. (설계 JSON 스키마·`assumptions`는 5장.)

```python
def make_design(requirement: str, *, limiter, feedback: dict = None) -> dict:
    """① 26B에게 요구사항을 주고 파일구조 JSON을 받는다.
    프롬프트에 반드시 박을 것: "불명확한 부분은 멋대로 기능을 늘리지 말고 둔 가정을 assumptions에 명시하라.
    시키지 않은 기능은 넣지 마라." + "코드 말고 구조만, JSON만, 마크다운 백틱 금지"(파싱 안정).
    feedback 있으면(①.5나 ③의 Issue dict) 그 title/detail을 '이 점을 고쳐라'로 함께 전달.
    추가로 누적 dead_end(known_failures) 리스트가 있으면 '이미 막힌 길이니 재시도 금지'로 프롬프트 상단에 박는다
    (부정 제약 — 8장). provides의 각 항목에 input_schema·output_schema(데이터 모양)도 적게 한다(5장),
    단 도메인 로직·구현 규칙은 넣지 말 것(JSON DSL화 금지 — 5장).
    반환: 설계 JSON(5장, assumptions·acceptance_criteria 포함). 파싱 실패 시 재요청."""

def review_design(requirement: str, design: dict, *, limiter) -> dict:
    """①.5 31B에게 요구사항+설계를 주고 '구현 가능? 모순 없나? 요구사항 빠뜨렸나?'를 판정.
    데이터 계약도 본다: depends_on으로 엮인 파일들의 input/output_schema가 서로 맞물리나(5장).
    반환: {"ok": bool, "issue": <공통 Issue> or None}. ok=False면 verdict는 보통 "design"(①로 되돌림).
    issue.title/detail이 make_design의 feedback으로 그대로 들어간다."""

def design_stage(requirement: str, *, limiter, max_rounds: int = 3) -> dict:
    """①↔①.5 루프: make_design → review_design → ok면 반환, 아니면 issue를 feedback으로 재설계.
    max_rounds 안에 통과 못 하면 마지막 설계 + 실패표시 반환(상위가 사람에게 보고)."""
```

**인터페이스**: 입력=요구사항 문자열(되돌림 시 feedback=③의 Issue, verdict=“design”). 출력=설계 JSON dict → 3번이 받아 파일별 코드를 만든다.

**단독 테스트**: `design_stage("간단한 CLI 할일 관리. 추가/삭제/완료/목록 4기능.", limiter=...)` → files에 storage/todo/main류, provides에 시그니처, depends_on 채워졌나. assumptions에 메운 가정이 명시됐나(멋대로 안 부풀렸나). 모순 있는 설계를 일부러 넣으면 review_design이 ok=False+issue를 주나.

**주의점**: JSON 파싱은 방어적으로(백틱 섞여도 추출, 실패 시 재요청). `truncated`면 설계가 잘린 것 → 재요청. max_rounds로 무한 왕복 방지.

## 코딩 (②) — `coding.py`

확정된 설계 JSON을 받아 **파일별로 코드 생성**(31B). 이 프로젝트의 급소인 컨텍스트 관리가 여기 있다 — 파일 B를 짤 때 다른 파일의 **전체 코드가 아니라 시그니처만** 준다(5장).

```python
def build_context(target_file: dict, design: dict) -> str:
    """target을 짤 때 31B에게 줄 컨텍스트 문자열: target의 name·role·provides(구현할 것)
    + depends_on에 적힌 다른 파일들의 provides(시그니처 + input/output_schema만! 코드 아님)
    + entry_point 여부. 스키마까지 줘야 데이터 계약 불일치(User vs {"name":str})를 막는다(5장).
    비유: 동료 코드를 다 읽는 게 아니라 API 문서(함수 명세+데이터 모양)만 보고 짜는 것(5장)."""

def code_one_file(target_file: dict, design: dict, *, limiter, feedback: dict = None) -> str:
    """② 31B에게 build_context + '이 파일 코드를 완성하라'를 주고 코드를 받는다.
    feedback 있으면(③에서 verdict="code"로 온 Issue) 그 title/detail을 함께('이 파일의 이 점을 고쳐라').
    누적 dead_end(known_failures) 리스트가 있으면 '이 접근들은 이미 실패, 재시도 금지'로 프롬프트 상단에 박는다
    (부정 제약 — 8장). next_guideline이 있으면 권하는 방향으로 함께 전달.
    반환: 파일 코드 문자열. truncated면 '큰 파일 쪼개기' 신호(13장)."""

def coding_stage(design: dict, *, limiter, parallel: bool = True) -> dict:
    """설계의 files 전체를 코딩. depends_on 순서 존중(의존 대상 먼저).
    parallel=True면 워커 병렬 — 토대 limiter 슬롯이 자연히 줄 세움(횟수만 경쟁).
    반환: {파일명: 코드문자열} dict."""
```

**인터페이스**: 입력=설계 JSON. 출력=`{파일명: 코드}` dict → 8장(검토)이 받아 정합성 검토하고 도커에서 실행.

**단독 테스트**: 설계 JSON을 `coding_stage`에 넣어 파일 수만큼 코드가 나오고 각 파일이 자기 provides 함수를 실제 정의했나. `build_context`를 단독 호출해 다른 파일 전체 코드가 안 샜는지(시그니처만) 확인.

**주의점**: **시그니처만 공유가 핵심** — build_context에 다른 파일 전체 코드가 새면 입력이 길어져 Gemma4 약점에 정통으로 맞는다(3·5장). 병렬은 limiter가 슬롯을 막아주나 결과 취합 시 dict 갱신은 lock 보호. `truncated`면 쪼개기 신호지만 지금은 감지·보고만(13장).

## 검토(③) + 도커 실행(④) — `review.py` · `runner.py`

생성된 코드를 설계와 대조해 정합성 검토(③, 26B), 통과하면 도커에서 실측(④). 문제가 나오면 설계/코드/환경으로 갈라 되돌린다. **③ 완전 구현, ④ 골격+계약만**(도커가 뱉는 에러 형태는 돌려봐야 알기에 짐작으로 안 채움 — 7장).

> **★ v0 경계(9.5장 v0 원칙과의 관계).** 2부는 *완성 시스템*의 시그니처 명세이지, *첫 구현(v0)*의 범위가 아니다. v0는 planner·coder·runner·observer·logger만이고(9.5장), 아래 `check_consistency`(AI 정합성 판정)와 `judge_error`(AI 원인 판정)는 **v0에 넣지 않는다** — 둘 다 *해석하는 AI*라 v0 원칙 4번(“AI 판정은 v0 밖”) 대상이다. 이것들은 0단계 로그가 “실패가 H1b(coder가 계약 깸)에 몰린다”를 보여줘 **호명할 때** 켜진다(H1b 실패 → check_consistency, H1a 실패 → review_design). v0에서 코드 정합성은 *판정*하지 않고 observer(ruff/mypy)가 *관측*만 한다 — 즉 아래 두 함수에서 라우팅·verdict를 떼고 “사실 출력”만 남긴 게 v0 observer다(9.5장 runs.jsonl 소절). 이 박스가 1부 9.5장과 2부를 잇는다: 2부의 풍부한 명세를 “지금 다 구현”이 아니라 “데이터가 부를 때 하나씩”으로 읽어야 한다.

> ③의 검증 원칙·계단식 파이프라인(Rule Engine→ruff/mypy→pytest→26B)·에러 3계층·안전장치는 **8장**에 있다. 여기 두 함수는 그 두뇌의 코드 시그니처다 — 둘 다 공통 Issue를 반환해, 5번 루프는 한 형식만 보고 verdict로 분기한다.

```python
# review.py (완전 구현)
def check_consistency(design: dict, codes: dict, *, limiter) -> dict:
    """③ 26B에게 설계 JSON + 코드들을 주고 대조: 'A가 호출한 함수를 B가 정의했나?
    설계의 provides/depends_on과 코드가 일치하나?'
    반환: {"ok": bool, "issue": <공통 Issue> or None}.
      ok=True → (skip_docker 아니면) 도커로 넘김 / ok=False → verdict로 ①(2로) 또는 ②(3으로, issue.target)."""

def judge_error(design: dict, codes: dict, run_result: dict, *, limiter) -> dict:
    """④ 도커 실행 실패 시 26B에게 원인 판정. 실측 에러(run_result['stderr'])를 근거로
    design/code/environment를 가른다(에러 3계층은 8장).
    반환: {"ok": bool, "issue": <공통 Issue>, "dead_end": str, "next_guideline": str}.
      ok는 보통 False — verdict로 되돌림 방향만 결정.
      dead_end       = 이번에 실패한 접근(다음 루프에 '재시도 금지'로 누적, 8장 known_failures).
      next_guideline = 다음 코딩/설계 모델에 줄 지침(부정 제약 + 권하는 방향).
      별도 요약 AI를 두지 않고 이 한 호출에서 함께 뽑는다(8장 MEMORY UPDATE 2단).
    세부 규칙은 도커로 진짜 에러를 받아본 뒤 다듬는다(13장 트리거 A)."""
```

```python
# runner.py (★ 골격+계약만, 알맹이 비움)
def run_in_docker(codes: dict, requirements: list, entry_point: str, *, timeout: int = 60) -> dict:
    """④ 생성 파일들을 격리 컨테이너에서 실제 실행, 결과 실측 반환.
    === 블랙박스 계약 (확정 — 도커 본문을 채워도 이 입출력은 유지. 7장) ===
    [입력] codes={파일명:코드}, requirements=설치 목록, entry_point=시작 파일, timeout=제한초.
    [출력] {"success":bool, "exit_code":int, "stdout":str, "stderr":str(judge_error 연료),
            "timed_out":bool, "stage":"build"|"run"}
    [본문 개념 — 지금은 NotImplementedError]
      1) 임시 폴더에 codes 쓰기 2) requirements 설치 3) 깨끗한 컨테이너에서 entry_point 실행(timeout)
      4) stdout/stderr/exit_code/타임아웃 수집 5) 컨테이너 폐기(매번 새 환경)
    """
    raise NotImplementedError("도커 본문은 나중 — 위 블랙박스 계약은 확정")

def looks_alive(run_result: dict, expected: str = None) -> bool:
    """'에러 없이 종료 ≠ 정상 동작' 보강(7·9장). expected가 주어지면 stdout에 그게 찍혔는지로
    한 겹 더 검증. 가능성 1층 판정에 쓴다. 도커 결과 형태 확정 후 다듬음."""
```

**인터페이스**: 입력=설계 JSON + 코드 dict + requirements/entry_point. 출력(되돌림)=공통 Issue → verdict가 “design”이면 2번 `design_stage`, “code”면 3번 `code_one_file`(issue.target). 통과하면 5번이 묶어 내보냄. **skip_docker 분기**: True면 ③ `check_consistency`까지만 하고 ④는 건너뜀 → ③ 통과를 통과로 간주. 이 플래그는 5번 `run_aaa`가 켠다(도커 본문을 채우면 떼는 게 목표).

**단독 테스트**: review.py(지금 가능) — 일치하는 설계+코드 → ok=True / 일부러 ‘A가 부른 함수를 B가 안 만든’ 코드 → ok=False, verdict=‘code’, target 맞나. runner.py(도커 본문 후) — 정상 todo → success=True, stdout에 기대출력 / import 빠진 코드 → success=False, stage=‘run’, stderr에 ImportError.

**주의점**: 지금 `run_in_docker`는 `NotImplementedError` — 계약(입출력 dict)만 의지처로 나머지를 짠다. judge_error 세부 규칙은 실제 에러 본 뒤(13장). “성공” 정의에 `looks_alive` 보강 잊지 말 것(빈 프로그램 함정).

## 출력(⑤) + 전체 루프 — `output.py` · `loop.py`

검토·실행을 통과한 결과를 **전체 파일 + requirements + 실행법**으로 묶고(⑤), 토대~검토를 **하나의 루프**로 잇는다(aaa의 바깥 껍데기).

```python
# output.py
def bundle_output(codes: dict, requirements: list, entry_point: str, run_result: dict, out_dir: str) -> str:
    """통과한 결과물을 out_dir에 묶는다: 각 파일을 .py로 저장 + requirements.txt + README(설치·실행
    명령, entry_point) + run_result 요약(실측 성공/출력)을 README에 첨부. 반환: 묶인 폴더 경로."""
```

```python
# loop.py (본체)
def run_aaa(requirement: str, *, task_type: str = None, out_dir: str = "./output",
            max_loops: int = 5, skip_docker: bool = True) -> dict:
    """aaa 전체 루프(skip_docker 기본 True — 도커 본문이 빌 동안의 기본 모드). 순서:
      1) limiter = Limiter() (usage 자동)
         state = TaskState(task_id, expected_type=task_type, stage="design", attempt=0)
                 # task_type=의도한 결합도 칸 A~E — planner엔 넘기지 않는다(적응 방지, 9.5장)
      2) design = design_stage(requirement, limiter=...)         # ①+①.5
         state.designed_files = [f["name"] for f in design["files"]]   # H1a 원재료(9.5장)
      3) codes  = coding_stage(design, limiter=...)              # ②
         state.generated_files = codes                          # H1b 원재료(designed와 대조, 9.5장)
      4) review = check_consistency(design, codes, limiter=...)  # ③
         ok=False면 review['issue'].verdict 따라 ①(2로) 또는 ②(3으로) 되돌림 후 재시도
      5) skip_docker=False일 때만: run = run_in_docker(codes, ...)  # ④
           실패면 judge_error로 판정 → verdict 따라 되돌림 후 재시도
         skip_docker=True면 ④ 건너뛰고 ③ 통과를 통과로 간주.
      6) 통과하면 bundle_output(...)                             # ⑤
      매 단계 state.stage/attempt 갱신·save, 실패 시 state.add_failure·add_dead_end(8장 TaskState).
      되돌림 때 state.dead_end를 feedback과 함께 다음 단계로 주입(부정 제약).
      되돌림은 max_loops 안에서(state.attempt로 카운트). 초과하면 state.status='stopped'로 멈추고
      현재 상태 + 실패이력 반환(가능성 판단용, 사람이 본다 — 디스코드 통지 9·10장).
      ★ 종료 시(성공·실패·중단 가리지 않고): state.dump_to_dataset("runs.jsonl")로 결과 한 줄 적재(9.5장),
        그리고 디스코드로 판단형 통지(final_result·loop_count·failed_stage·retry_value·recommended_action,
        본문 20줄·전체로그 첨부 — 10장).
    반환: {"status": "ok"|"failed", "out_dir":..., "history":[...각 라운드 Issue 포함...]}"""
```

**인터페이스**: `loop.py`가 토대~검토 함수들을 순서대로 호출하는 유일한 조립처. 되돌림 라우팅은 공통 Issue의 verdict로 2번(design)/3번(code) 분기. ④가 `NotImplementedError`라 `skip_docker=True`가 기본 — 도커 없이 ①~③+⑤를 끝까지 돌려 감을 잡는다(9장 2층 수동 판정). 같은 이슈 3회 반복 시 설계 롤백 등 안전장치는 8장.

**단독 테스트**: `run_aaa("간단한 CLI 할일 관리. 추가/삭제/완료/목록.", skip_docker=True)` → 설계→코딩→정합성검토까지 돌고 output 폴더에 .py들+requirements+README가 묶이나. 그 결과물을 2층 4항목으로 직접 본다(9장). 도커 본문 후 skip_docker=False로 실측 연결.

**주의점**: max_loops 상한 필수(되돌림 무한 방지). 초과는 실패가 아니라 **가능성 판단 데이터** — history를 사람이 본다(10장 통지). 목표는 100%가 아니라 “고쳐 쓸 만한가”(9장). loop의 목적은 “완벽 통과”가 아니라 “사람이 볼 결과물을 안정적으로 뱉기”.

### 구현 순서

토대 → 설계 → 코딩 → 검토(review.py만, runner는 계약만) → 출력+루프(skip_docker=True로 ①~③+⑤ 끝까지) → (도커 본문 후) runner.py 채우고 skip_docker=False. 각 단계는 단독 테스트를 통과시키고 다음으로.

## 인터페이스 정합성 표 (★ 멀티파일이 맞물리는지 한눈에)

이 시스템의 급소는 “A가 부른 함수를 B가 진짜 정의했나”(파일 간 정합성). 아래 표로 각 단계의 입력이 앞 단계의 출력과 같은 모양인지 못 박는다 — 코드를 짤 때 이 “받는 것/주는 것”을 계약으로 삼는다.

### 데이터가 단계 사이를 흐르는 모양

### 되돌림 라우팅 (Issue.verdict 하나로 갈림)

|issue.verdict  |누구에게                              |무엇으로                    |
|---------------|----------------------------------|------------------------|
|`"design"`     |설계 `make_design(feedback=issue)`  |설계 다시 (요구사항+이전설계+지적)    |
|`"code"`       |코딩 `code_one_file(feedback=issue)`|해당 `issue.target` 파일만 다시|
|`"environment"`|requirements/도커 설정                |환경 교정 (8장 에러 3계층)       |

### 정합성 자가 체크리스트 (각 단계 완성 후 확인)

1. **출력 모양이 표와 같나** — 예: `design_stage`가 `assumptions` 포함 JSON을 주나(코딩이 그걸 기대).
1. **받는 쪽이 그 모양을 그대로 읽나** — 예: `coding_stage`가 `files[].depends_on`으로 시그니처+스키마 컨텍스트를 짜나.
1. **데이터 계약(스키마)이 맞물리나** — depends_on으로 엮인 두 파일의 output_schema↔input_schema가 같은 모양인가(이름만 같고 데이터가 어긋나는 함정 — 5장).
1. **되돌림은 Issue 형식 하나뿐인가** — 설계·검토 어디서 나오든 `{ok, issue}` 동일. 루프가 verdict만 보고 분기.
1. **TaskState가 단일 출처인가** — 단계·시도·실패를 AI 머릿속이 아니라 TaskState가 들고, 매 단계 save 되나(8장).
1. **토대 `call_model` 외 직접 호출이 없나** — limiter/usage를 우회하면 한도 관리가 샌다.
1. **skip_docker=True에서 ④를 안 부르나** — `run_in_docker`가 `NotImplementedError`라 호출되면 터진다.
